In a [previous post](#/blog?id=safety-align-as-world-reasoner), I argued that robust safety alignment requires models to become world reasoners: capable of predicting how their actions change states of the world, rather than pattern-matching over prohibited linguistic forms. That post ended by pointing toward minimal world-interacting agents as a practical intermediate path, somewhere between the pure text world and fully embodied interaction. This post is about one concrete instance of that path.

The fundamental problem with supervising safety in a pure text environment is that the supervision signal and the thing being supervised are the same medium. Consequences are described, not produced. LLM-as-judge systems evaluate linguistic plausibility. Human labelers are inconsistent on borderline cases. Refusal templates get paraphrased around. None of these signals measure whether a model has correctly reasoned about what would happen next; they measure whether its output resembles safety-trained text. A model that understands what safety-trained text looks like can optimize toward that form without changing anything about its underlying reasoning — which is part of why adversarial prompting works so reliably. It does not challenge the model's values; it reroutes around its pattern matcher.

Code has a property that natural language fundamentally lacks: executability. When a model writes code and the code runs, the environment responds. A function returns a value. A simulation advances a state. The output is a consequence, not a description of one, minimal and contained but real within its environment. This is categorically different feedback from asking another language model whether an output seemed dangerous.

Code is also verifiable and interactive. Whether a program achieves its stated goal can be determined by running it. A model working in a code environment must act, observe the result, and continue, participating in a feedback loop where actions have deterministic effects. The evaluation does not depend on a judge's priors or a labeler's intuitions; the environment itself provides the verdict. Together, these properties make code a minimal world: an environment where supervision can be grounded in something external to the model's own output distribution.

The objection that comes naturally is that harmful requests are not coding problems. "How do I synthesize this compound?" has no direct translation into code, and forcing one would be artificial. But the argument is not about direct translation. Harmful events almost never unfold in a single step; they progress through chains of enabling actions, each advancing the process toward its outcome. Many of those intermediate steps, even when the overall goal has nothing to do with programming, involve structured, verifiable computational tasks: querying a source, filtering and organizing information, connecting one component to another. A harmful goal may not be expressible as code, but a causally critical step toward that goal often can be. Information processing is where capability translates into real-world effect: a model that can efficiently locate, structure, and route the right inputs is materially more useful to a harmful process than one that cannot, regardless of whether its outputs ever explicitly discuss harm.

When a model is placed at one of those intermediate steps and completes it, the completion is a different kind of evidence from a harmful-sounding text response. The model has taken an action that materially advances a harmful process, producing a working artifact rather than dangerous-looking text. The question shifts from what the model said to what the model did. A model that declines to execute a causally critical step is doing something different from a model that produces text flagging the request as harmful. The first is a refusal at the level of capability; the second is a surface output that a sufficiently capable model can work around.

This is what event-level supervision means. The unit of analysis is the event: the causal progression from a harmful intent through a series of enabling steps toward an outcome. Text-based evaluation cannot register whether a model is contributing to that progression, because it can only compare outputs to other outputs. Code, by introducing a substrate where causal contribution can be verified, provides a different kind of ground truth. Training against this kind of signal would create pressure on models to refuse at the level of actual contribution to harm, not merely at the level of output form.

The broader principle connects back to the world reasoner argument: alignment methods that rely entirely on text to supervise text are optimizing within the same medium they are trying to constrain. Code, as a minimal world, introduces at least one point where that loop opens up and supervision can touch something real. The space of verifiable intermediate steps is itself an open question — not every harmful goal has a clear computational proxy, and identifying which ones do requires understanding the structure of harmful processes. But the direction is the right one. Building more of these bridges, and understanding where they hold and where they break, is where I think the next progress in consequence-grounded safety supervision will come from.
