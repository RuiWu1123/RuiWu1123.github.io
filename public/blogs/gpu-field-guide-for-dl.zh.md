我们大多数训练模型的人，其实从来不需要真正去思考 GPU 这个东西。你写下 `model.to("cuda")`，写下 `loss.backward()`，几周之后一个数字降下来了。底层的硬件是一个黑箱，偶尔会甩给你一句 `CUDA out of memory`，除此之外它安安静静地待在你看不见的地方。

这篇文章，写给这个黑箱不再"安安静静待着"的那一刻——当你想弄明白，为什么你的训练一步要 400ms 而不是 40ms；为什么把 batch size 翻倍居然没有拖慢任何东西；为什么你精心搭建的 8 卡并行任务，跑起来的效果却像是只有 3 张卡在干活；又或者，mixed precision training 这件事本身，究竟为什么会存在。这些问题都不需要你会写 CUDA kernel。但它们都需要你对 GPU 到底"是什么"有一个能用的心智模型——这正是本文想要建立的东西：从零开始，但专门为深度学习研究者会遇到的问题而搭建，而不是一篇面向大众的科普。如果你已经知道什么是 warp，那第一部分对你来说大概只是复习；请耐心看到 roofline 模型和多卡那两节，那才是真正撑起直觉的部分。

## 1. 之后所有内容都由这一个想法衍生而来

CPU 和 GPU 都是由晶体管构成的，都在做算术运算。两者的区别，几乎完全在于"它们各自在为什么目标做优化"。

CPU 为 **延迟（latency）** 而优化：让一条指令流尽可能快地跑完，包括它所有的分支、依赖关系和不可预测性。为了做到这一点，CPU 的一个核心把大部分晶体管都花在了跟算术运算本身毫无关系的地方——分支预测、乱序执行、大容量缓存、投机执行——目的是让单条控制流跑得尽可能接近物理极限。一颗 CPU 上通常也就 8 到 128 个这样的核心。

GPU 为 **吞吐量（throughput）** 而优化：它假设你手头有成千上万份几乎完全相同、彼此独立的工作要做（一帧画面里的每个像素，或者一个 tensor 里的每个元素），它并不在乎其中任何"一份"工作跑多久，只要能保证每秒钟把海量的这类工作推送出去。所以 GPU 相对而言在控制逻辑和分支预测上花的晶体管很少，几乎把所有面积都投入到了纯粹的算术单元上。你得到的不是少量复杂的核心，而是成千上万个简单的核心。

![CPU vs GPU chip layout](blogs/images/cpu-vs-gpu.svg?v=3)
^CPU 把大部分芯片面积用于控制逻辑与缓存，从而让单条指令流跑得更快；GPU 则把大部分面积用于算术单元（ALU），让同一条指令同时在成千上万个线程上执行。

这也是为什么"GPU 有 16,896 个 core，CPU 只有 16 个"这种说法虽然没错，但如果只看字面意思会很误导人——GPU 上的一个 core（通常叫 CUDA core）是一个极其微小、简单的 ALU，只能做一次标量乘加。它跟 CPU 的一个 core 完全不是一回事，后者本身就是一个完整、独立的处理器。真正合理的对比方式，不是"core 对 core"，而是"在一个具备足够并行度的负载下，这块芯片每秒能完成多少有效的算术运算"——而按这个标准衡量，在深度学习这种典型负载下（同样的少数几种运算——matmul、加法、乘法、一个非线性函数——被无差别地施加在数百万个独立元素上），GPU 能领先 1 到 2 个数量级。

这也正是接下来需要一直记住的约束：GPU 之所以快，*正是因为* 它假设你的工作是规整且并行的。你的代码里每一次出现依赖数据的分支、细碎的顺序依赖、或者不规则的内存访问，本质上都是在和"让 GPU 之所以成为 GPU"的那个前提对抗。

## 2. Grid、Block、Thread、Warp：任务究竟是怎么被调度的

要真正用上这种算术吞吐能力，你需要一种编程模型，能够表达"对独立的数据做同一件事，重复海量次"。用 CUDA 的术语来说，这个模型是一个层级结构：

一次 **kernel launch** 会启动一个 **grid**。一个 grid 被划分成若干 **thread block**（简称 block）。每个 block 又被划分成若干 **thread**。当你写一段 PyTorch 代码时，你完全看不到这些细节——框架以及它底层调用的 kernel（来自 cuDNN、cuBLAS，或者自定义的 Triton/CUDA kernel）会替你选好 grid 和 block 的维度——但这个层级结构的形状，仍然完全决定了你的性能表现。

真正在硬件层面起决定作用的细节是这个：thread 并不是一个一个地跑，但也不是完全各跑各的。它们以 **32 个为一组** 运行，这一组被称为一个 **warp**，warp 里的 32 个 thread 在同一时刻执行 *同一条指令*，只是作用在不同的数据上（这种执行模型被称为 SIMT——single instruction, multiple threads）。如果你写了一个 `if` 语句，warp 里 16 个 thread 走 `true` 分支，另外 16 个走 `false` 分支，那么这个 warp 并不会把两个分支并行跑完——它会先跑 `true` 分支，同时把另外 16 个 thread 屏蔽掉；再跑 `false` 分支，屏蔽掉前 16 个。这被称为 **warp divergence（warp 分歧）**，也是在自定义 kernel 中，理论吞吐量在不知不觉中大量流失的常见原因之一。

从物理结构上看，GPU 芯片就是一个由 **Streaming Multiprocessor（SM）** 组成的阵列——一块 H100 上有 132 个 SM。每个 SM 都有自己的 warp scheduler、自己的 CUDA core、自己的 Tensor Core、自己的寄存器堆（register file），以及一块片上 shared memory。关键在于：**一整个 thread block 会被调度到恰好一个 SM 上**，并且在它的整个生命周期内都待在那个 SM 上——它绝不会在执行过程中迁移。同一个 SM 上可以同时驻留多个 block（这正是 "occupancy" 这个词所指的东西），但单个 block 永远不会跨越两个 SM。

![Inside one Streaming Multiprocessor](blogs/images/gpu-sm-anatomy.svg?v=3)
^每个 SM 内部包含若干 warp scheduler、成组的 CUDA core（标量/向量算术）与 Tensor Core（矩阵算术）、一个寄存器堆，以及 shared memory / L1 cache。一个 block 在它的整个生命周期里，完完全全生活在同一个 SM 上。

一个 SM 上同时能驻留多少个 block，取决于每个 block 对这个 SM 有限资源——寄存器、shared memory，以及最大可驻留 warp 数（在近几代架构上通常是 64 个 warp，也就是 2048 个 thread）——的占用程度。一个每个 thread 用更少寄存器、或者每个 block 用更少 shared memory 的 kernel，就能在同一个 SM 上塞进更多 block，这通常意味着更好的延迟隐藏能力（当某个 warp 在等待一次很慢的内存读取时，scheduler 可以切换到另一个已经就绪、可以计算的 warp），并且在一定范围内，也意味着更高的吞吐量。这就是"occupancy"这个在几乎每一个 GPU profiler 里都会出现的术语的本质。

在实践中真正重要的结论是：**发射的 block 太少，会导致芯片没被用满**（有些 SM 处于空闲状态）；而 **block 很小、但每个 thread 占用资源很多**，则会导致每个 SM 内部的 occupancy 很低，尽管每个 SM 表面上"都有活干"。这两种失效模式，在 `nvidia-smi` 里都会表现为"我的 GPU 利用率很低"，但背后的原因截然不同——这也正是为什么这一个孤立的利用率数字，本身是一个很差的诊断工具。

## 3. 内存层级：真正限制你的东西

这里有一个足以重新框定本文其余大部分内容的事实：**在几乎所有现代加速器上，相对而言，搬运数据的代价远远高于对数据做算术运算的代价。** 一次 fused-multiply-add 几乎是免费的。而把它所需要的两个数从内存里错误的位置取出来，其代价可能是算术运算本身的 100 到 1000 倍还多。深度学习的性能优化工作，很大一部分其实都是在尊重这个事实。

GPU（和 CPU 一样）也有一套内存层级：多个层次的存储，在容量和速度之间做取舍。

![GPU memory hierarchy](blogs/images/memory-hierarchy.svg?v=3)
^这个金字塔每往下走一层，容量大致会增加 5–20 倍，而带宽大致会下降 5–20 倍。快的 kernel，会尽量在这个金字塔里靠上的位置复用数据，而不是一次次跑到下面去重新读取。

其中几层特别值得单独说一下，因为它们直接对应着你会不断遇到的一些概念：

**寄存器（Registers）** 是某个 thread 私有的，从功能上讲访问几乎是免费的——但每个 thread 能用的寄存器数量非常有限，而每个 thread 用的寄存器太多，正是降低 occupancy 的主要原因之一（每个 thread 占用的寄存器越多，一个 SM 上能塞下的 block 就越少）。

**Shared memory / L1** 是片上存储，由一个 block 内的所有 thread 共享，大致比访问 device memory 快 20 到 30 倍。它是 kernel 作者手里，把一个 memory-bound 的操作变成 compute-bound 操作的主要工具：把一块数据从 device memory 加载进 shared memory 一次，之后让很多 thread 从这里反复复用它，而不是每个 thread 各自重新去取一遍。这正是矩阵乘法在 GPU 上能跑得快的核心原因，我们会在第 5 节回过头来细说。

**HBM（High Bandwidth Memory）** 就是大家平时说的"GPU 显存"，也是 `nvidia-smi` 里报出来的那个数字，同时也是显存不足（OOM）报错时所指的东西。按 DRAM 的标准来说它已经很快了——H100 大约有 3.35 TB/s 的 HBM3 带宽——但它仍然比片上存储慢一到两个数量级，而且你的模型接触到的每一个 tensor、每一份 activation、每一份权重，在两次 kernel 调用之间都得存在这里。

**Host（CPU）DRAM，经由 PCIe** 是你的 dataloader 产出的数据在被拷贝到 GPU 之前所停留的地方，也是 CPU 端 tensor 生活的地方。PCIe Gen5 单方向大约能提供 60–64 GB/s 的带宽——比 HBM 带宽低两个数量级。这正是为什么一个慢的 dataloader 能在悄无声息间把你的 GPU 利用率封顶在 40% 左右：GPU 跑完一步之后就闲下来了，等着下一个 batch 从 CPU 那侧、经过这座相对狭窄的桥，被搬运过来。

**经由 NVLink 或网络访问的远程 GPU 内存**，就是 *另一块* GPU 上的 activation、gradient 或者 KV-cache，从你这块 GPU 的视角看起来的样子——又比上一层慢了一到两个数量级，这是第 6 节要讨论的主题。

一个有用的思维习惯是：每当你看到一个很慢的操作，先问"这个操作需要从这个金字塔的多深处，搬运多少字节的数据"，然后再问"它做了多少 FLOPs"。对相当大一部分深度学习负载来说，第二个问题的诚实答案其实是"没做多少"——而它之所以慢，完全可以用第一个问题来解释。

## 4. Tensor Core，以及为什么精度是一个系统层面的决策，而不只是数值层面的

CUDA core 做的是标量或小向量的 fused-multiply-add：每条指令、每个 thread，一次乘法加一次加法。从 Volta 架构（2017 年）开始，NVIDIA 在每个 SM 里都加入了第二种算术单元：**Tensor Core**，它能在一条指令里计算一次小规模的 *矩阵* 乘加（可以理解成类似 4×4 乘以 4×4 的矩阵乘法，累加进一个 4×4 的结果，全部在一条指令里完成）。由于深度学习本质上就是海量的矩阵乘法堆叠而成，Tensor Core 恰好能在那些主导了训练和推理的运算上，提供比 CUDA core 高出大约一个数量级的吞吐量——前提是你的数据格式是 Tensor Core 支持的格式。

正是最后这一句话，让"精度"从一个纯粹的数值问题，变成了一个系统层面的问题。Tensor Core 通常在更低精度的数据格式上跑得最快，因为更窄的数字意味着同样的硅片面积和内存带宽，每个周期能塞进更多的数。在 H100 上，dense（非稀疏）峰值吞吐大致上，每往下换一种格式就会翻一番：

| 格式 | 峰值 dense 吞吐（H100 SXM5，约） | 典型用途 |
|---|---|---|
| FP32（CUDA core） | ~67 TFLOPs | 参照基准 / 非 Tensor Core 运算 |
| TF32（Tensor Core） | ~495 TFLOPs | 给 FP32 写的训练代码"白送"的加速 |
| BF16 / FP16（Tensor Core） | ~990 TFLOPs | 标准的 mixed-precision 训练 |
| FP8（Tensor Core） | ~1979 TFLOPs | 较新的训练/推理场景，需要更谨慎的 scaling |

（以上数字是 NVIDIA 公开的 H100 SXM5 dense、非稀疏峰值数据；更新一代的 Blackwell B200 把这条曲线又往前推了一大截，拥有大约 8 TB/s 的 HBM3e 带宽、约 5 PFLOPs 的 dense BF16 吞吐，以及一种面向推理的原生 FP4 格式。）

这正是 mixed-precision training 之所以存在的原因：它根本不是一个为了炫技而发明的数值技巧，而是硬件如此重奖"更窄格式"这一事实的直接后果。围绕它的那些数值层面的复杂性——loss scaling、在 FP32 里保留一份 master weights、选择动态范围更宽的 BF16 而不是 FP16——全部都是为了安全地拿到这 2 到 4 倍的硬件加速（以及按比例减少的内存流量，而根据第 3 节的内容，这一点往往比纯粹的 FLOPs 数字更重要）。FP8 训练把这个权衡又往前推了一步，需要相应更谨慎的 scaling；它通常不像 TF32 那样是"即插即用"式的免费午餐，至今仍是一个活跃的方法论研究方向，而不是一个可以随手切换的开关。

实际意义在于：如果你不确定一篇论文为什么要如此仔细地说明它使用的精度，或者一份训练方案为什么坚持把某些层（比如最终的 loss 计算，或者 normalization 的统计量）保留在更高精度上，答案通常正是这里所说的这层张力——更窄的格式是硬件真实给出的巨大加速，但代价是动态范围和精度上的损失，而你的数值方法必须足够稳健才能扛住这个代价。

## 5. Roofline 模型：你的 kernel 是被算力卡住，还是被数据搬运卡住

第 3 节和第 4 节其实都在指向同一个底层问题，而这个问题有一个标准的分析工具，可以给出精确答案：**roofline 模型**。

对于一个给定的操作，定义 **arithmetic intensity（算术强度）** 为它执行的浮点运算次数与搬运的内存字节数之比（FLOPs / byte）。每个操作能达到的峰值吞吐由下式给出：

**可达 TFLOPs/s = min( 峰值算力, arithmetic intensity × 峰值内存带宽 )**

在 arithmetic intensity 较低时，你处于 **memory-bound（受限于内存）** 状态：GPU 的算术单元大部分时间都在空闲，等待数据，你能达到的吞吐量随着你能用上的带宽线性增长。在 arithmetic intensity 较高时，你处于 **compute-bound（受限于算力）** 状态：内存能够轻松地把算术单元"喂饱"，你能达到的吞吐量被芯片的峰值 FLOPs 封顶，无论你还有多少富余的带宽都无济于事。算力和带宽这两个限制因素同等重要的那个交叉点，被称为 **ridge point（临界点）**，它只取决于芯片本身（峰值算力 ÷ 峰值带宽），跟你写的代码无关。

一个 transformer 内部的各个具体操作，在这条曲线上所处的位置可能天差地别。Elementwise 操作（加一个 bias、一个 ReLU、一次 residual add）每个元素只做一到两次 FLOPs，却需要从 HBM 里把这个元素读进来再写出去——arithmetic intensity 大约在 0.1 到 0.5 之间，深深地处于 memory-bound 的区域。Normalization 和 softmax 也只是略好一些。相比之下，一个规模合适、形状良好的大矩阵乘法，会像下一节要讲的那样，对每个加载进来的值反复复用很多次，其 arithmetic intensity 可以达到数百——牢牢地处于 compute-bound 区域，也正是 Tensor Core 真正大显身手的地方。

这一个想法就能解释很多原本令人困惑的经验事实：为什么把好几个小的 elementwise 操作 fuse 成一个 kernel（`torch.compile` 和手写的 fused kernel 都在做这件事）能带来很大的加速，而 FLOPs 的数量却 *完全没变*——你并不是做了更少的算术运算，而是减少了往返 HBM 的次数，而后者才是真正限制你的东西；为什么在长序列、小 batch 的场景下，attention 的实际速度相对于它的 FLOPs 数量而言会令人失望地慢——因为它是 memory-bound 的，真正主导耗时的是搬运 attention matrix 和 KV-cache，而不是矩阵乘法本身（这也是 FlashAttention 出现的很大一部分动机：它专门重新组织了计算过程，以避免把那些 memory-bound 的中间 tensor 真正物化出来）；以及为什么更大的 batch size 在墙钟时间/样本这个指标上，如此频繁地"几乎白送"——因为它通过让每次权重加载分摊到更多样本上，提高了 arithmetic intensity，把你进一步推向 compute-bound 区域,而在此之前，你其实一直在白白浪费掉一部分吞吐量。

## 6. 矩阵乘法：把这个最重要的操作映射到芯片上

由于矩阵乘法既是深度学习中 FLOPs 消耗的绝对主力，也是上面所有内容最清晰的一个具体案例，值得比"cuBLAS 会处理好这一切"再往下深挖一层，看看它到底是怎么被执行的。

一个朴素的矩阵乘法实现，会让每一个输出元素直接从 HBM 里读取 A 的整整一行和 B 的整整一列，计算一次点积，写出一个输出值。这种做法的 memory-bound 程度是灾难性的：A 的每个元素，会随着 B 的每一列被重新从 HBM 读取一次，反之亦然——产生了大量对最慢那一层内存的冗余访问。

真实的实现方式则会对这个问题做 **tiling（分块）**。输出矩阵 C 被切分成若干 tile，每个 tile 被分配给一个 thread block。这个 block 会协作式地把它需要的那一小条 A 和那一小条 B——只加载一次——搬进片上的 shared memory。从那之后，block 内的每个 thread 都通过反复读取 *shared memory*、而不是 HBM，来计算它负责的那部分输出，在被换出之前把每个加载进来的值反复复用很多次。

![Matmul tiling diagram](blogs/images/matmul-tiling.svg?v=3)
^把一块 A 和一块 B 加载进 shared memory 一次，之后为输出 tile 里的每一个元素反复复用它们，正是把矩阵乘法从一个 memory-bound 操作变成 compute-bound 操作的关键所在。

这正是上一节"arithmetic intensity"这个抽象概念背后具体的实现机制：tiling 把一次 HBM 加载，变成了很多次算术运算；而 tile 越大（这通常意味着你的 batch size 或者 hidden dimension 越大），每加载一个字节所能换来的复用次数就越多，你也就越接近芯片的峰值 FLOPs。这也直接解释了，为什么非常小的矩阵——比如 32 的 hidden dimension，或者 1 的 batch size——会长期性地跑不满一块 GPU：无论 kernel 调得多好，每个 tile 能获得的复用次数根本不够，不足以摊薄加载它的成本。

## 7. 多卡：互联方式决定了并行策略的天花板

前面所有内容说的都是单块 GPU。而在当下的规模上训练任何东西，都意味着要协调很多块 GPU，连接它们的物理拓扑绝不是一个无关紧要的细节——它直接决定了哪种并行策略真正能够 scale 起来。

在单个节点内部，现代系统通常通过 **NVLink**（每块 GPU 大约 900 GB/s 到 1.8 TB/s，取决于具体代数）连接 GPU，往往还会通过一个 **NVSwitch**，让节点内的每一块 GPU 都拥有通往其他任意一块 GPU 的全带宽路径。而在节点之间，GPU 通常通过 **InfiniBand** 或 RoCE 之类的网络结构通信，每块 GPU 大约 400 Gb/s（约合 50 GB/s）——这又是一个数量级的下降，这次是从节点内的 NVLink，降到了节点间的网络。

![Multi-GPU topology](blogs/images/multi-gpu-topology.svg?v=3)
^每往外走一跳，带宽大致都会下降一个数量级：片上 → HBM → NVLink（节点内） → 网络结构（跨节点）。

这种不对称性，正是不同并行策略之所以存在、并且以现在这种方式被组合使用的全部原因：

**数据并行（Data parallelism）**——每块 GPU 都持有一份完整的模型副本，处理 batch 里不同的一部分，每一步结束后再把各自算出的梯度做一次 all-reduce 求平均——通信频率相对较低，能够比较好地容忍较慢的跨节点链路。这也是为什么它是最容易跨越大量节点扩展的策略。

**张量并行（Tensor parallelism）**——把单个权重矩阵切分到多块 GPU 上，使得单独一层的计算本身就是分布式的——需要在 *每一层内部* 通信，一次前向加反向传播里可能要通信很多次。这种量级的流量，需要依赖快速、低延迟的 NVLink 域；如果把张量并行放到较慢的跨节点链路上运行，正是"线性扩展"曲线一旦跨越多个节点就不再线性的一个非常具体、也非常常见的原因。

**流水线并行（Pipeline parallelism）**——把模型的各层切分到不同 GPU 上，不同 GPU 负责同一次前向/反向传播的不同阶段——通信频率比张量并行低（只需要在阶段边界传递 activation），因此能更好地容忍跨节点链路，代价是众所周知的"pipeline bubble"空闲时间。

在实践中，大规模训练任务通常会混合使用这几种策略——张量并行留在节点内部（走 NVLink），流水线并行和/或数据并行跨越节点（去容忍网络带宽）——原因正是上面那张拓扑图所决定的：不同的距离上，能够承受的取舍是不一样的。当有人说某个训练任务在超过一定节点数之后扩展效率就不行了，第一件值得检查的事，就是看看是不是有一种假设了 NVLink 级别带宽的并行策略，被意外地放到网络结构上去跑了。

## 8. 汇总：训练变慢时的一份简短排查指南

以上所有小节都不只是纯理论——它们可以归纳成一份相当简短的清单，覆盖了大多数"为什么我的训练这么慢"的排查场景：

**GPU 真的在忙吗？** `nvidia-smi` 的利用率告诉你的，是"某一时刻是否有 kernel 在 GPU 上跑"，而不是"这个 kernel 是否在高效地做有用的工作"。一个慢的 dataloader，或者训练循环里不必要的 CPU–GPU 同步点（热循环里一个不起眼的 `.item()`、`.cpu()`，或者 `print(tensor)`），常常会表现为"平均利用率看起来还不错"，而实际上 GPU 是在一阵阵短暂的忙碌之间不断地空闲着。

**这个 kernel 是 memory-bound 还是 compute-bound？** 第 5 节 roofline 的问题。Profiler（Nsight Compute，或者 PyTorch 自带的 profiler）会直接告诉你实际达到的 FLOPs 和实际达到的带宽利用率；如果带宽利用率很高、FLOPs 利用率很低，那么无论换多"快"的 GPU，效果都远不如减少内存流量（算子融合、`torch.compile`、避免不必要的中间 tensor）。

**Occupancy 合理吗？** 第 2 节的问题——你发射的并行工作量是否足够，block size 有没有浪费寄存器或 shared memory，是否真的能把芯片填满？这一点主要跟手写或自定义 kernel 有关，标准的 PyTorch 算子通常已经调得相当合理了。

**你用的精度，是硬件真正重奖的那种吗？** 第 4 节的问题——mixed precision（BF16/FP16，或者在你的训练方案支持的情况下用 FP8）往往是可选改动里性价比最高的那一个，正因为硬件本就是被设计成如此重奖它。

**如果是多卡场景，你的并行策略和拓扑匹配吗？** 第 6 节的问题——张量并行的流量跑到了慢的跨节点链路上，而不是留在快速的 NVLink 域内，是多节点扩展效果不及预期的最常见原因之一。

**你是不是其实受限于显存容量，而不是算力，只是它表现成了 OOM 而不是变慢？** 决定速度的同一套内存层级，也同时决定着容量——activation 占用的内存、optimizer states，以及 gradient checkpointing 的权衡（用重新计算 activation 来换取 HBM 容量，而不是把它们都存下来），都是第 3 节那个金字塔的具体体现，只是这次看的是容量这根轴，而不是带宽那根轴。

以上这些都不需要你自己去写一个 CUDA kernel。但一旦"算力 vs. 带宽"、"warp vs. occupancy"、"NVLink vs. 网络"不再是陌生的术语，而是变成你面对一个"训练效果不如纸面 FLOPs 数字所暗示的那样好"的任务时，会下意识去调用的心智模型，上面这些问题就都从"猜"变成了"可以回答"。

*本文中出现的数字（H100/B200 规格、带宽、TFLOPs）均来自 NVIDIA 公开的 datasheet，目的是传达正确的数量级，以及各个量之间正确的相对关系，而不是作为可引用的精确规格表——具体数字会因 SKU、散热方案、驱动/固件版本而有所不同，也会随着每一代新硬件的发布而变化。真正经久不衰的，是 SIMT 执行模型、内存层级、roofline 模型、以及互联拓扑这些概念本身，而不是附着在它们身上的某个具体数字。*
