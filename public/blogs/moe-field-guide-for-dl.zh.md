Mixture-of-Experts（MoE）是 2026 年几乎每一个前沿语言模型底层都在用的架构，但"MoE"这三个字母背后，实际涵盖的设计差异比听起来的要大得多。单说"MoE"本身，几乎说明不了这个模型到底是怎么路由 token 的、怎么维持专家之间的均衡的、专家池里到底有多少个专家、真正激活的又有多少个，甚至也说明不了"专家"内部的计算现在是不是还长得像一个普通的前馈模块。这篇文章讲的就是这个标签背后的具体机制：一层现代 MoE 到底在做什么、它的每一个组成部分历史上是怎么来的，以及路由策略、门控函数、负载均衡方式、专家粒度、潜空间压缩、按 token 自适应算力这几个具体的设计杠杆，前沿实验室们现在都在怎么拉动它们、把这个架构继续往前推。

## 一、一层现代 MoE 长什么样

把所有厂商特有的细节都去掉，2026 年这一代的 MoE 层大致是这样运作的：一个 token 进来，一个小型路由网络给每个专家打一个"匹配度"分数，得分最高的 top-K 个专家真正跑一遍前馈计算，而在现在几乎所有的设计里，还有一个额外的共享专家，不管路由器怎么判断，它都会对每个 token 无条件运行。所有输出按路由器给出的权重加权合并，就是这一层的输出。

![一层现代 MoE 的解剖图](blogs/images/moe-layer-anatomy.svg?v=1)
^一个 token，一层：每个路由专家都是候选项，真正跑起来的只有寥寥几个。共享专家是唯一的例外，它永远会被激活。

这个设计的全部魅力都藏在两个数字之间的落差里。**总参数量** 取决于专家池里有多少专家，这就是厂商放进头条的数字（"2.8 万亿参数"）。**激活参数量** 取决于每个 token 真正跑了多少专家，也就是 top-K 加上共享专家，这才是真正决定推理成本的数字。只要 K 相对于整个专家池保持得足够小，一个总参数量巨大的模型，单个 token 的推理成本依然可以很低。总量和激活量之间的这道落差，就是 MoE 存在的全部理由，下面提到的每一个设计选择，说到底都是在尝试把这道落差拉得更大，同时又不伤害模型质量。

## 二、这些设计到底从哪来：九年的谱系

这篇文章里几乎每一个提到的机制，要么是某个更老想法的变体，要么是对某一篇 2024 年论文的直接延续。在讲最近这些新变化之前，值得先把这条线捋一遍。这个想法比它通常被归入的深度学习时代还要古老：**Jacobs 等人，1991 年** 最早提出了 mixture-of-experts 的想法，让不同的子网络在任务的不同部分上各自专精，那时候连类似现代 transformer 的东西都还不存在。下面梳理的，是这个想法被搬到大型神经序列模型上之后的谱系，也就是大多数人说"MoE"时，实际指的那段历史。

![MoE 设计的九年演变](blogs/images/moe-lineage-timeline.svg?v=1)
^2026 年大多数前沿 MoE 设计都能追溯到 2024 年的同一个分叉点：DeepSeekMoE 的"细粒度专家 + 共享专家"配方。

**Shazeer 等人，2017 年** 为深度学习引入了现代意义上的稀疏门控 MoE 层：一个带噪声的 top-k softmax 路由器，配合一个辅助损失来维持专家使用的均衡。这基本上是后面所有工作的共同祖先。

**GShard，2020 年** 把这个想法扩展到了真正跨硬件做专家并行训练的 transformer 上：把专家切分到不同设备，用 all-to-all 通信把 token 送到持有其对应专家的设备。标准的负载均衡辅助损失公式也是从这里来的，容量因子（每个专家能接收的 token 数硬上限，超出部分会被丢弃）也是从这里进入标准工具箱的。

**Switch Transformer 和 GLaM，2021 年** 从同一个基础出发走向了不同方向：Switch Transformer 把路由简化成 top-1（每个 token 只用一个专家），证明了即便在这种极端设置下 MoE 依然能稳定训练；GLaM 则用 top-2 路由把参数规模做到 1.2 万亿，每个 token 只激活其中很小一部分，这是"把总参数量和计算成本解耦"这件事一次早期而干净的示范。

**ST-MoE 和 expert-choice 路由，2022 年** 解决了一个训练稳定性问题（路由器的 logits 可能无限增长导致训练不稳定，用一个 router z-loss 来解决），并探索了一套完全不同的范式：不再是每个 token 挑选它的 top-K 专家，而是让每个专家挑选属于它的 top-tokens，这样从结构上就保证了完美的负载均衡，但也让因果、自回归生成变得更复杂。

**Mixtral，2024 年** 与其说带来了新的 MoE 机制，不如说证明了整套方法可以被训练出来，并作为一个强力的开源权重模型发布：采用相当标准的 softmax 路由器，做 8 选 2 的路由。从此，MoE 在任何想做开源模型的团队眼里，变成了一个主流、不再显得另类的架构选择。

**DeepSeekMoE，2024 年** 才是后面几乎所有内容真正的分叉点。它的第一个贡献是细粒度专家切分：不用少量的大专家，而是用大量的小专家，这给了路由器在给一个 token 分配容量时多得多的组合灵活性。第二个贡献是把一个或多个共享专家隔离出来，让它们无条件处理每一个 token，这样通用的、大家都需要的计算就不必在几十个路由专家里被反复重复学习。

这个分叉点也是"共享专家"这个问题的起点，接下来两年里，不同的实验室对这个问题给出了不同的答案，不只是 DeepSeek 自己后续的模型，阿里巴巴的 Qwen 系列在这个问题上的立场也反复调转过。第七节会详细讲这部分。

还有两个更晚近、范围更窄的想法，虽然不是 DeepSeekMoE 这条主线的直系祖先，但对后面的内容同样重要：**Hash Layers，Roller 等人，2021 年** 证明了完全可以用一个固定的、确定性的哈希函数来路由 token，而不需要任何可学习的门控，用路由质量换取绝对均衡的负载和零路由网络开销（这种权衡具体的代价有多大，第四节会细讲）。而 **Mixture-of-Depths，2024 年** 让 transformer 可以根据一个逐 token 学到的决策，选择让 token 穿过或绕过整个计算模块，是"让部分 token 跳过真正计算"这个想法最直接的概念祖先，此外还有另一项 2024 年提出的、用于加速专家并行通信的技术 ScMoE。

## 三、DeepSeek-V3：MoE 真正的成名之作

如果说 DeepSeekMoE（2024 年）是那个分叉点，那 DeepSeek-V3（2024 年末）就是把这个分叉变成行业默认设置的那次发布。它完全没有改动"细粒度专家 + 共享专家"这套底层配方，这部分原封不动。但在这套配方之上，它又打包上了三个路由和训练层面的决策，此后大部分同行要么直接照搬，要么明确以此为参照另辟蹊径。

最引人注目的改动是 **免辅助损失负载均衡**：DeepSeek-V3 没有再加一个和语言建模目标竞争梯度信号的独立损失项，而是给每个专家一个可学习的偏置项，这个偏置项只加进路由决策（也就是决定哪些专家被选中），而不会加进用来合并专家输出的权重里：

$$
\text{score}_i = g_i + b_i, \qquad \text{weight}_i = g_i
$$

$$
b_i \leftarrow \begin{cases} b_i - \gamma & i \text{ 过载} \\[2pt] b_i + \gamma & i \text{ 欠载} \end{cases}
$$

这里，$g_i$ 是路由器给专家 $i$ 打出的原始匹配度分数，计算方式和第五节讲的门控函数一致。$b_i$ 是专家 $i$ 独有的一个可学习偏置，每训练一步调整一次，调整幅度是一个很小的固定步长 $\gamma$：如果专家 $i$ 在上一步负载过高就往下调，负载不足就往上调。这个包含偏置的分数，只用来决定 top-K 选择，也就是决定这个 token 最终由哪几个专家来处理。之后用来合并这些专家输出的权重，只用 $g_i$ 本身，不包含偏置，所以偏置只能影响"谁被选中"，永远不会影响"选中之后这个专家的输出算多少"。均衡完全通过这个逐步调整来实现，不需要再调一个互相竞争的损失系数。

在此之上，还叠加了一个权重很小的序列级均衡损失：一种传统的 GShard 式损失项，专门用来抓住单条序列内部的不均衡，这种不均衡如果只靠偏置项（在更粗的、批次级别的粒度上更新）是有可能漏掉的。偏置机制才是主要的均衡信号，序列级损失更像是一道保险。（第六节会把这个机制和这个领域其他的均衡方案放在一起比较。）

配套打包的还有另外两个决策。**节点限制路由（node-limited routing）** 把每个 token 的路由候选限制在数量有限的几个计算节点上的专家，而不是整个集群，从而给单个 token 能产生的 all-to-all 流量设了一个上限。以及一个和主目标（预测下一个 token）一起训练的 **多 token 预测（MTP）模块**，提供更密集的训练信号，也是投机解码（speculative decoding）天然可以借用的一个组件。

单独看这四个想法，没有哪一个读起来像是对 GShard 时代 MoE 的一次剧烈背离。真正让 DeepSeek-V3 成为 MoE"成名之作"的，是这个组合恰好在这个行业需要这样一个参照点的时刻，装进了一个强力、且公开详细披露的模型里，而接下来十八个月里同行的反应，比这套机制本身更能说明这一点。GLM-5.2 几乎原封不动地沿用了这套 MoE 配方。DeepSeek 自己的 V4 把这个偏置机制当作基线，只做了扩展。就连 Qwen 独立的谱系，它自己从来没有采用过这个偏置项机制，最终也收敛到了同一个底层判断上：学习式路由需要某种针对路由坍缩的"保险"。它和 DeepSeek-V3 出于同样的理由，选择了共享专家作为这份保险。三个不同的实验室，三种不同的具体机制，背后是同一个发现。

## 四、三种路由方式：Hash、学习式与 Sinkhorn

到目前为止，前面提到的一切都假定了一个前提：学习式路由，一个可训练的门控网络给专家打分，梯度下降慢慢塑造这些分数。这是行业里的默认做法，但它是一个选择，而不是唯一的选项，把它和其他几种方式放在一起比较，可以说明为什么这个领域会在负载均衡上花这么多工程精力，而不是干脆挑一种结构上天然就均衡的路由机制。

![三种给 token 分配专家的方式](blogs/images/moe-routing-comparison.svg?v=1)
^同样 8 个 token，同样 4 个专家，三种不同的分配规则。只有一种同时做到了均衡负载，又真的看了 token 的内容。

$$
\text{expert}(x) = h(\text{id}) \bmod N
$$

$$
g_i = \text{router}(x)\cdot e_i, \qquad \text{selected} = \operatorname{top\text{-}}k(g_1,\dots,g_N)
$$

$$
A \leftarrow \text{row\_normalize}(A), \qquad A \leftarrow \text{col\_normalize}(A)
$$

$$
\text{selected} = \operatorname{top\text{-}}k(A_i)
$$

在这三种方式里，$x$ 都是 token 的表示，$e_i$ 都是专家 $i$ 学到的键向量，$N$ 是专家池的大小。对 Hash 路由来说，$h(\text{id})$ 是对 token 位置或标识做的一个固定哈希，完全没有可学习参数。对学习式路由来说，$g_i$ 是路由器训练出来的、对专家 $i$ 的匹配度分数，top-$k$ 会选出得分最高的 $k$ 个专家（具体用什么函数算出 $g_i$，是 softmax、sigmoid 还是其他变体，第五节会细讲）。对 Sinkhorn 路由来说，$A$ 是整个批次的"token × 专家"匹配度矩阵，行归一化和列归一化会交替重复若干轮，之后 top-$k$ 选择作用在归一化后的矩阵 $A_i$ 上，但之后的合并权重依然用原始的、未归一化的 $g_i$，而不是 $A_i$，原因下面会讲。

Hash 路由是三者里最简单的一个：它完全不看 token 内容，只看它在批次里的位置，所以永远不会有两个 token 争抢同一个专家，也永远不会有专家被饿死。但这份保证同时也是它的天花板。因为分配结果和 token 到底是什么毫无关系，路由器压根没有信号可以利用，相比一个计算量对齐的稠密模型，在 16 个专家的规模下测得的提升大约封顶在 1.5%，而且随着专家数增加几乎不再改善，因为专家再多，对一个对内容视而不见的分配规则来说也没有更多信号可用。

学习式路由走的是相反的路线：因为门控是在真实的 token 表示上训练出来的，它可以真正做到专精（在 16 个专家的规模下大约有 4% 的改进，是 Hash 路由的两倍还多），但它的训练目标本身，并不会自动阻止它把所有 token 都送去同样那几个专家。这种失效模式，也就是路由坍缩，往往在网络最早和最后几层表现得最严重，这正是共享专家（第七节）或者一个显式的均衡损失（第六节）想要补上的那个缺口。

Sinkhorn 路由试图同时拿到两者的好处。在做 top-k 选择之前，整个批次的"token × 专家"匹配度矩阵会交替沿行和列做归一化：按行归一化让每个 token 的打分加起来等于 1，按列归一化让每个专家收到的负载加起来也相等，重复若干轮之后，矩阵会收敛到一个既对每个 token 打分合理、又让每个专家负载均衡的状态，从而在每一层（而不只是平均意义上）都达到 Hash 级别的均衡。代价是这种接近完美的均衡，限制了单个专家相对于普通学习式路由能达到的专精程度。而且还有一个具体的实现陷阱：由于 Sinkhorn 归一化本身是一个固定的、不带梯度的过程，如果直接把它的输出当作合并权重来用，会悄悄把路由器和训练信号完全断开。实践中的修复方式，是只用 Sinkhorn 归一化后的矩阵来决定哪些专家被选中，而合并权重依然用原始的、未归一化的 logits 来计算，这样即便选择过程是由一个独立的、不可微的流程来做均衡的，路由器本身依然在持续学习。

学习式路由的坍缩风险，也是一个更小、更冷门的实现细节反而变得格外重要的地方。具体到 top-1 路由：标准的门控权重归一化方式（用一个 token 的门控值除以它所选专家的门控值之和）在只选一个专家的情况下，会退化成一个数除以它自己，结果永远是 1。这会悄悄把路由器从语言建模损失那里本该收到的梯度清零，最终只有负载均衡损失的梯度能传到路由器，路由器可以把负载均衡学得完美无缺，却完全没学到哪个专家真正适合哪个 token。一种已有文献记录的修复方式，是加一个"空专家"（null expert）：一个从不真正计算任何东西的虚拟选项，不额外花参数，也不额外花算力，但能给路由器一个真实的对比对象，让它的 top-1 选择有东西可以对照，从而恢复来自主损失的有效梯度信号。这是一个只需改一行代码就能修好的 bug，但单看路由器自身的使用率统计，永远发现不了它，因为不管路由器有没有真的学到东西，均衡到完美的专家使用率看起来都是一样的。

## 五、门控函数：从 softmax 到 sigmoid 及其变体

路由器最核心的工作，是把一个 token 的表示转换成对每个专家的打分，而用哪种函数来打分，已经悄悄变成了整个架构里影响最大的设计选择之一。

$$
\textbf{Softmax:}\quad g_i = \frac{\exp(x\cdot e_i)}{\sum_j \exp(x\cdot e_j)}
$$

$$
\textbf{Sigmoid:}\quad g_i = \frac{1}{1+\exp(-x\cdot e_i)}
$$

$$
\textbf{Sqrt(Softplus):}\quad g_i = \sqrt{\log(1+\exp(x\cdot e_i))}
$$

在每一种情况下，$x$ 都是 token 的隐藏表示，$e_i$ 都是专家 $i$ 学到的键向量，所以 $x\cdot e_i$ 就是两者的点积相似度，也就是原始 logit。Softmax 会在整个 $N$ 个专家的集合上一起做归一化，强制所有 $N$ 个分数加起来等于 1：抬高专家 $i$ 的 logit，必然会压低其他每一个 $g_j$。Sigmoid 和 Sqrt(Softplus) 则各自对每个 logit 独立套用一个函数，不做跨专家的归一化，所以一个专家的分数变化，不会对任何其他专家的分数产生任何影响。

![在共享预算里竞争，还是各自独立打分](blogs/images/moe-gating-comparison.svg?v=1)
^同样 5 个专家，同样的起始 logits，只有 3 号专家的 logit 被调高。在 softmax 下，其他所有专家的打分都得相应缩水；在 sigmoid 下，别的专家分数完全不变。

早期的 MoE 工作（Shazeer、GShard、Switch Transformer、Mixtral）用的都是跨所有候选专家的 softmax，让每个专家去争抢一份共享的概率预算：抬高一个专家的分数，必然会压低其他所有专家的分数，不管这对那些专家来说是不是真的合理。DeepSeek-V3 换成了 sigmoid，让每个专家对着一个固定阈值打分，而不是和专家池里的其他专家比较。这个区别在专家数只有个位数的时候基本无关紧要，但专家数涨到成百上千之后影响会大得多：在 256 或 896 个专家的规模下，一次 softmax 调整会被摊薄到整个专家池里，而 sigmoid 的独立性不管 N 多大都保持不变。

sigmoid 这个选择基本上延续了下来。[GLM-5.2](https://www.zhipuai.cn/zh/research/161) 原封不动地保留了它，[DeepSeek-V4](https://arxiv.org/pdf/2606.19348) 也保持了同样的"独立打分"思路，只是把具体的函数换成了 Sqrt(Softplus)，论文里说这相较 V3 的 sigmoid 只是一个细微调整，不是性质上的改变。[MAI-Thinking-1](https://microsoft.ai/pdf/mai-thinking-1.pdf) 是唯一一个明显的反例，回退到了普通的 softmax，而且是在 token 原始的、未压缩的表示上计算的，尽管真正的专家计算发生在压缩后的潜空间里（第八节）。值得注意的是，MAI-Thinking-1 也是这里唯一一个没有让每一层都用 MoE 的架构，而是把稠密前馈块和稀疏 MoE 块交替排布，这就带来一种可能性：门控函数的选择，或许在某种程度上和 MoE 在整个网络里被应用得有多密集是耦合在一起的，而不是一个放之四海而皆准的改进。[Kimi K3](https://www.kimi.com/blog/kimi-k3) 的门控函数，在其发布材料里没有披露。值得一提的是，Qwen 这条谱系在历代模型里都没有离开过 softmax，提醒我们 sigmoid 的优势是真实的，但并不是放之四海而皆准的。

## 六、负载均衡：从辅助损失到偏置项，再到分位数

路由器的训练目标本身并不会自然而然地阻止它把所有 token 都送去同样那几个专家、然后把其余专家全部闲置，这种失效模式通常被称为路由坍缩（routing collapse）。如何在不伤害模型质量的前提下防止它，是 MoE 设计里过去两年被重新发明次数最多的部分。

![DeepSeek-V3 基于偏置项的均衡闭环](blogs/images/moe-balancing-loop.svg?v=1)
^没有另外的竞争损失项。路由器自己的偏置根据上一步测得的负载自我调整，一步一步进行。

$$
L_{\text{aux}} = \alpha \cdot N \cdot \sum_{i=1}^N f_i \, P_i
$$

$$
L_z = \frac{1}{B}\sum_{\text{tokens}} \Big(\log \sum_i \exp(\text{logit}_i)\Big)^{2}
$$

这里，$f_i$ 是一个批次里实际被送到专家 $i$ 的 token 所占的比例，是一个硬计数，本身不可微。$P_i$ 是同一个批次里，路由器分配给专家 $i$ 的平均路由概率，是软的、可微的。把两者乘起来，就让 GShard 的辅助损失 $L_{\text{aux}}$ 有了一个梯度，专门去压低那些"既被频繁选中、打分又很自信"的专家，其中 $\alpha$ 是一个可调系数，$N$ 是专家池大小。在 router z-loss $L_z$ 里，$B$ 是批次大小，求和遍历每个 token 的 logit 向量；对 log-sum-exp 项取平方，直接惩罚过大的 logit，让 softmax 在数值上保持稳定，和负载是否均衡完全是两回事。DeepSeek-V3 完全抛弃了这两种做法，改用第三节讲过的按专家偏置机制。

GShard 最早的方案，就是这个加进训练目标里的辅助损失，后来 Switch Transformer 和 ST-MoE 又补充了 router z-loss，专门用来提升 logit 的数值稳定性。DeepSeek-V3 完全用它的偏置机制取代了辅助损失，均衡负载的同时，完全不需要和语言建模损失去竞争任何梯度。DeepSeek-V4 保留了这个偏置机制，但在此之上又叠加了一个小的序列级均衡损失，专门用来抓住单条长序列内部的不均衡，这种不均衡只有在按整个训练批次平均之后才会看起来正常，考虑到 V4 面向百万 token 上下文的目标，这一点尤其相关。

MAI-Thinking-1 走了另一个方向，用的是相当传统的 GShard 式辅助损失，而不是偏置项。论文报告说，一旦 $f_i$ 和 $P_i$ 都在整个训练批次范围内做全局聚合，而不是按 micro-batch 分别计算，它的表现就和基于偏置项的方法差不多。Qwen3 大约提前一年，独立得出了一个惊人相似的结论。标准的辅助损失通常是按 micro-batch 分别计算的，如果某个 micro-batch 恰好在领域上有偏斜（比如因为数据加载器凑批次的方式，刚好全是代码），这个损失就会实际惩罚路由器做了一件本该做对的事：把代码 token 送去代码专精的专家，强行要求每个 micro-batch 内部都呈现出人为的均匀分布。Qwen 的修复方式，在一篇专门的文章里做了说明，并在一篇关于负载均衡损失实现细节的配套论文里被正式化：在计算这个损失之前，先把每个专家的选择频率在所有 micro-batch 之间同步起来，这样均衡约束作用的对象就变成了整个语料库，而不是其中每一个单独的切片。单个 micro-batch 可以自由保持领域偏斜，整体聚合起来依然均衡。两个实验室，相隔一年，各自独立收敛到同一个修复方式：均衡损失的聚合粒度，看起来比具体建立在它之上的公式更重要。

Kimi K3 是离 DeepSeek-V3 默认设置最远的一个。它完全不用偏置项，而是用了 Moonshot 称之为 Quantile Balancing 的方法，直接从路由器自身打分分布的分位数推导出专家分配，同时消除了启发式的偏置更新规则和一个需要单独调节的均衡超参数。在 Kimi K3 896 选 16 这种稀疏度下，微小的路由偏差比在 DeepSeek-V3 256 选 8 下有更大的累积空间，所以一家前沿实验室想少一个容易调错的超参数，是很合理的诉求。几种不同的均衡机制在大约十八个月之内接连发布，而公开材料里没有一份能干净、受控地证明其中哪一种全面胜过其他几种，很大程度上是因为每一种都是在一个特定的架构里调出来的（不同的粒度、不同的门控函数、不同程度的 dropless），单凭公开报告很难从中提炼出一个真正对等的比较。

## 七、粒度与共享专家问题

$$
\textbf{粗粒度:}\quad N \text{ 个专家，top-}K\text{ 激活}
$$

$$
\textbf{细粒度:}\quad N\!\cdot\! m \text{ 个专家，top-}(K\!\cdot\! m)\text{ 激活}
$$

这里，$N$ 是粗粒度基线里的专家数，$m$ 是 DeepSeekMoE 引入的切分因子。每个原始专家的前馈模块被切成 $m$ 个等大的小块，所以细粒度专家池有 $N\cdot m$ 个专家，而不是 $N$ 个，top-$K$ 也相应变成 top-$(K\cdot m)$，以保持激活的比例不变。两种做法下每个 token 花掉的计算量完全一样，因为总的宽度没有变，只是分配方式变了，但路由器现在可以从一个大得多的组合空间里挑选：在 $N=8$、$K=2$ 的情况下只有寥寥几种可能的组合，而 $N\cdot m=64$、$K\cdot m=16$ 时，可能的组合数会一下涨到百万级别。

![同样的总容量，切法不一样](blogs/images/moe-granularity-comparison.svg?v=1)
^8 个专家、top-2，只有 4 种可能的组合。把同样的总宽度切成 64 份、top-16，能填满这份预算的方式就爆炸式增长到了百万级：计算量不变，专精的方式却多了很多。

DeepSeekMoE 的细粒度专家这个想法，已经不再像它配方里"共享专家"那一半那样，还处于被激烈讨论的状态，它是这篇文章里几乎所有内容共同的出发点。Kimi K3 把这个想法推得比这篇文章里任何其他模型都更远，896 个专家，选 top-16。

至于共享专家该不该和这些细粒度专家搭配使用，是一个真正还没有定论的开放问题，而不是一个已经尘埃落定的默认选项。阿里巴巴的 Qwen 团队围绕这个问题做了持续时间最长、也最公开的一场实验，而且自己的答案反转了两次。

![Qwen 的共享专家反转](blogs/images/moe-qwen-lineage.svg?v=1)
^同一条谱系，得出了相反的结论，然后又部分收回：Qwen 连续一年多不断加共享专家，随后完全去掉，再之后一款更新的模型据报道又把它加了回来。

Qwen1.5-MoE（2024 年 3 月）和 Qwen2-MoE（2024 年年中）都把细粒度路由专家和若干个常驻共享专家结合了起来：先是 4 个共享加 60 个路由、top-4，后来是 8 个共享加 64 个路由、top-8。Qwen2 自己的技术报告说得很直白：共享专家可以"在各类任务中都发挥作用，同时把其他专家留给特定路由场景里有选择性地使用"，引用的正是驱动 DeepSeekMoE 和 DeepSeek-V3 采用共享专家的那个路由坍缩担忧。Qwen3（2025 年 5 月）随后把方向完全调转了过来：它的技术报告直接写道，"与 Qwen2.5-MoE 不同，Qwen3-MoE 的设计排除了共享专家"，转而依靠第六节讲过的全局批次负载均衡损失，作为防范路由坍缩的保险。关于 Qwen3-Next（2025 年 9 月）的二手报道，尚未在第一手技术报告里得到确认，描述了共享专家又重新出现，和 512 个专家里的 10 个路由专家搭配使用，如果属实，这会让 Qwen 自己的历史在大约十八个月内完成一次完整的往返：共享专家先加入，然后去掉，据报道之后又加了回来。

MAI-Thinking-1 是从另一个角度走到"不要共享专家"这个结论的：团队测试过共享专家之后发现，它只在每一层都用 MoE 的架构里才有帮助。由于 MAI-Thinking-1 是把稠密层和稀疏层交替排布，而不是这样做，共享专家在他们自己的消融实验里没有带来可测量的收益，最终发布的模型也没有用它。[LongCat-2.0](https://longcat.chat/blog/longcat-2.0/) 则完全绕开了这个二选一的问题，把路由专家划分成三个按任务划分的组，Agent、Reasoning、Interaction，按任务分别门控，而不是把整个专家池当成一组无差别的、可以互换的通才专家外加一个共享专家。

## 八、把计算本身压缩掉：LatentMoE

$$
z = W_{\text{down}}\, x, \qquad y_i = \text{expert}_i(z), \qquad \text{out} = W_{\text{up}}\, y_i
$$

$$
\text{cost} \propto N\ell,\ K\ell
$$

这里，$x$ 是 token 的完整维度表示，维度为 $d$；$z$ 是它经过一个可学习的下投影矩阵 $W_{\text{down}}$ 之后，投影到的一个更小的潜空间，维度为 $\ell < d$。专家计算完全在这个压缩后的空间里进行，之后一个可学习的上投影矩阵 $W_{\text{up}}$ 再把结果展开回维度 $d$，才和这一层其余部分的输出合并。由于单个专家的计算量和 all-to-all 通信的数据量，都是按 $\ell$ 而不是 $d$ 来算的，用 top-K 跑 $N$ 个专家的成本，就变成上面那样按 $\ell$ 缩放，而不是通常的 $Nd, Kd$。$d$ 和 $\ell$ 之间的这个差值，正是可以用来"购买"更大专家池或更大 top-K 的那份预算，而总成本并不增加。

![LatentMoE：压缩、计算、展开](blogs/images/moe-latentmoe-flow.svg?v=1)
^标准 MoE 里，每个专家、每一次 all-to-all 都要花掉完整的 token 维度 d。LatentMoE 花的是压缩后的维度 l，省下来的差额被重新投入到更大的专家池里。

到目前为止提到的想法，都没有动专家计算本身，只是在它周围调整路由或均衡方式。LatentMoE 则直接改变了计算本身。它由 NVIDIA 在 2026 年年初率先提出，已经用在自家 Nemotron-3 Super 和 Ultra 模型上。

MAI-Thinking-1 直接采用了这个思路，从更早的 192 个专家、top-4 配置成长到了 512 个专家、top-8，正是在 LatentMoE 让这个更大的配置在通信上变得负担得起之后才实现的。它还完全采用 dropless 路由，也就是完全没有 token 丢弃容量上限这回事，论文说这个选择是因为一旦引入硬性容量上限，他们在均衡消融实验里就会看到与容量相关的不一致结果。Kimi K3 的"Stable LatentMoE"看起来是同一个潜空间压缩家族被用在了更极致的粒度上（896 个专家、top-16），不过 Moonshot 自己的材料没有像 MAI-Thinking-1 的报告那样，公布具体的压缩比例（$\ell$ 相对 $d$ 的比值）或专家扩展比例。

## 九、让计算本身变得自适应

以上所有想法依然假定一个固定的 top-K：不管是一个标点符号还是一个真正困难的推理步骤，每个 token 激活的专家数量都是一样的。LongCat-2.0 直接打破了这个假设。它的专家池里包含了美团称之为 Zero-Computation Expert 的东西：这些"专家"完全不做任何计算，只是原样把输入透传出去。路由器会对每个 token 决定要用多少个真正的专家、多少个零计算专家。一个简单的 token 可能几乎全部被路由到零计算专家，几乎不花什么算力；一个困难的 token 才会调用更多真正的专家算力。这是一种按 token 自适应的算力预算，而不是固定的 top-K，从概念上比 DeepSeekMoE 这条谱系更接近 Mixture-of-Depths 在 2024 年提出的"让 token 完全跳过计算模块"的想法：同一个想法，被用在了专家维度上，而不是深度维度上。

这也是唯一一个还留有真实、未解答的开放问题的部分。美团还没有公布这套设计具体的负载均衡机制，而零计算专家这个机制本身就带来一个明显的问题：在训练时，是什么阻止了路由器为了在简单 token 上把 loss 降到最低，而把所有东西都路由到零计算专家。目前公开的材料还没有回答这个问题。

## 十、系统层面：路由到底要花多少通信成本

每一个被路由的 token，都得实实在在地通过互联，"跑"到持有它所选专家的那块加速卡上，这个成本只会随着专家数量涨到成百上千而变得更紧迫（这篇[分布式训练文章](#/blog?id=distributed-training-for-dl)更详细地讲过这种 all-to-all 通信模式）。上面提到的好几个选择，都是从不同角度对同一个成本问题给出的回应。DeepSeek-V4 里 Hash 路由的复活，也就是确定性、无需学习的门控，专门用在前三层 MoE 上，取代了原本的稠密 FFN 块，用路由质量换取了这几层里绝对均衡的负载和零路由网络通信开销。LatentMoE 压缩后的潜空间，直接缩小了 all-to-all 交换中需要搬运的数据量，这正是让 MAI-Thinking-1 和 Kimi K3 一开始就能负担得起更大专家池的那个杠杆。而 DeepSeek-V4 另一项系统层面的贡献，以 MegaMoE 之名开源的流水线式专家并行内核，把专家并行 MoE 的 dispatch、计算、combine 三个阶段按专家"波次"重叠执行，而不是顺序执行，相比一个可比的重叠方案 Comet 报告了明显的加速。LongCat 的 ScMoE 骨干同样是一个系统驱动的选择，一种专门用来提升专家并行吞吐量的 shortcut-connected 设计。这些都不像 sigmoid 门控或细粒度专家那样是质量层面的提升，它们是让这些质量提升真正能负担得起运行成本的工程手段。

另外，DeepSeek-V4 还引入了 Anticipatory Routing，这是一项训练稳定性技术，而非通信技术：当模型检测到 loss 尖峰（论文明确把这归因于 MoE 层内部的异常激活值）时，会开始用路由器自身权重稍微滞后的一份拷贝来计算路由决策，而不是用当前权重，把路由器的更新速度和模型其余部分的更新速度解耦，直到不稳定状态过去。

## 十二、哪些已经收敛，哪些还在争论

细粒度路由专家在 DeepSeekMoE 发表之后大约两年内，就不再是一个被激烈讨论的选择了，它是这整篇文章的共同出发点。Sigmoid 系门控已经在大多数（但并非全部）前沿实验室里取代了 softmax，MAI-Thinking-1 回退到 softmax 是 2026 年唯一一个明确的例外，而 Qwen 整条谱系从一开始就没有离开过 softmax。

共享专家这个问题则恰恰相反，完全谈不上收敛。DeepSeek-V3、GLM-5.2、Kimi K3 都保留了共享专家；MAI-Thinking-1 测试过之后把它去掉了；Qwen 连续一年多不断加共享专家，在 Qwen3 里彻底去掉，再之后根据未经证实的报道，可能又在 Qwen3-Next 里把它加了回来。如果这篇文章里有哪一个设计维度是这个领域明显还没有想清楚的，那就是这一个。

负载均衡仍然明显每隔几个月就被重新发明一次：一个可学习的按专家偏置项、同样的偏置项再加一个序列级损失、一个完全按全局批次聚合的传统辅助损失（Qwen 和 MAI-Thinking-1 各自独立得出）、以及完全不需要单独超参数的分位数推导分配，这几种不同的答案在大约十八个月之内接连发布，而公开记录里没有任何受控比较能说明哪一种更胜一筹。这些方法背后，其实都建立在第四节讲过的同一个三选一之上，也就是 Hash、学习式，还是 Sinkhorn 路由，而这篇文章里的每一个前沿模型，在这个层面上都做出了同样的选择（学习式路由，再加上某种"保险"机制），尽管它们在具体用哪种保险机制上分歧很大。

在均衡机制这一层之上，最重要的一个分歧点，在于两种不同的、拉大总参数量和激活参数量之间差距的方式。LatentMoE 保留了静态的 top-K，但通过把计算搬进压缩后的潜空间，降低了更大的 K 和更大的 N 的实际成本：专家更多、激活的也更多，但每一个都更便宜。LongCat-2.0 的零计算专家则保留了专家池名义上的大小不变，但让真正被激活的"实体"专家数量变成了一个动态的、取决于 token 的量。这两条路并不互斥，如果未来某份报告把两者结合起来，用潜空间压缩让更大的专家池变得负担得起，再用按 token 自适应的激活方式避免把这份预算平均花在不需要它的 token 上，也不会让人意外。就像任何针对一个快速演进的架构所做的快照一样，这里读起来像是已经收敛的部分，六个月后也可能会变得不一样。
