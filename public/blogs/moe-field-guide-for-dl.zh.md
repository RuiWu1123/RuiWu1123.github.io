Mixture-of-Experts（MoE）是 2026 年几乎每一个前沿语言模型底层都在用的架构，但"MoE"这三个字母背后，实际涵盖的设计差异比听起来的要大得多。单说"MoE"本身，几乎说明不了这个模型到底是怎么路由 token 的、怎么维持专家之间的均衡的、专家池里到底有多少个专家、真正激活的又有多少个，甚至也说明不了"专家"内部的计算现在是不是还长得像一个普通的前馈模块。这篇文章讲的就是这个标签背后的具体机制：一层现代 MoE 到底在做什么、它的每一个组成部分历史上是怎么来的，以及门控函数、负载均衡方式、专家粒度、潜空间压缩、按 token 自适应算力这几个具体的设计杠杆，前沿实验室们现在都在怎么拉动它们、把这个架构继续往前推——文中的具体证据来自几份 2026 年的模型报告，再加上它们共同建立在其上的那些经典 MoE 文献。

## 一、一层现代 MoE 长什么样

把所有厂商特有的细节都去掉，2026 年这一代的 MoE 层大致是这样运作的：一个 token 进来，一个小型路由网络给每个专家打一个"匹配度"分数，得分最高的 top-K 个专家真正跑一遍前馈计算，而且——在现在几乎所有的设计里——还有一个额外的共享专家，不管路由器怎么判断，它都会对每个 token 无条件运行。所有输出按路由器给出的权重加权合并，就是这一层的输出。

![一层现代 MoE 的解剖图](blogs/images/moe-layer-anatomy.svg?v=1)
^一个 token，一层：每个路由专家都是候选项，真正跑起来的只有寥寥几个——共享专家是唯一的例外，它永远会被激活。

这个设计的全部魅力都藏在两个数字之间的落差里。**总参数量** 取决于专家池里有多少专家——这就是厂商放进头条的数字（"2.8 万亿参数"）。**激活参数量** 取决于每个 token 真正跑了多少专家——top-K 加上共享专家——这才是真正决定推理成本的数字。只要 K 相对于整个专家池保持得足够小，一个总参数量巨大的模型，单个 token 的推理成本依然可以很低。总量和激活量之间的这道落差，就是 MoE 存在的全部理由，下面提到的每一个设计选择，说到底都是在尝试把这道落差拉得更大，同时又不伤害模型质量。

## 二、这些设计到底从哪来：九年的谱系

这篇文章里几乎每一个提到的机制，要么是某个更老想法的变体，要么是对某一篇 2024 年论文的直接延续——在讲最近这些新变化之前，值得先把这条线捋一遍。

![MoE 设计的九年演变](blogs/images/moe-lineage-timeline.svg?v=1)
^2026 年大多数前沿 MoE 设计都能追溯到 2024 年的同一个分叉点：DeepSeekMoE 的"细粒度专家 + 共享专家"配方。

**Shazeer 等人，2017 年** 为深度学习引入了现代意义上的稀疏门控 MoE 层：一个带噪声的 top-k softmax 路由器，配合一个辅助损失来维持专家使用的均衡。这基本上是后面所有工作的共同祖先。

**GShard，2020 年** 把这个想法扩展到了真正跨硬件做专家并行训练的 transformer 上——把专家切分到不同设备，用 all-to-all 通信把 token 送到持有其对应专家的设备。标准的负载均衡辅助损失公式也是从这里来的，容量因子（每个专家能接收的 token 数硬上限，超出部分会被丢弃）也是从这里进入标准工具箱的。

**Switch Transformer 和 GLaM，2021 年** 从同一个基础出发走向了不同方向：Switch Transformer 把路由简化成 top-1（每个 token 只用一个专家），证明了即便在这种极端设置下 MoE 依然能稳定训练；GLaM 则用 top-2 路由把参数规模做到 1.2 万亿，每个 token 只激活其中很小一部分——这是"把总参数量和计算成本解耦"这件事一次早期而干净的示范。

**ST-MoE 和 expert-choice 路由，2022 年** 解决了一个训练稳定性问题（路由器的 logits 可能无限增长导致训练不稳定，用一个 router z-loss 来解决），并探索了一套完全不同的范式——不再是每个 token 挑选它的 top-K 专家，而是让每个专家挑选属于它的 top-tokens，这样从结构上就保证了完美的负载均衡，但也让因果/自回归生成变得更复杂。

**Mixtral，2024 年** 与其说带来了新的 MoE 机制，不如说证明了整套方法可以被训练出来、并作为一个强力的开源权重模型发布——采用相当标准的 softmax 路由器，做 8 选 2 的路由——从此让 MoE 在任何想做开源模型的团队眼里，变成了一个主流、不再显得另类的架构选择。

**DeepSeekMoE，2024 年** 才是后面几乎所有内容真正的分叉点。它的两个贡献是：第一，细粒度专家切分——不用少量的大专家，而是用大量的小专家，这给了路由器在给一个 token 分配容量时多得多的组合灵活性；第二，把一个或多个共享专家隔离出来，让它们无条件处理每一个 token，这样通用的、大家都需要的计算就不必在几十个路由专家里被反复重复学习。

**DeepSeek-V3，2024 年末** 接着做出了三个改动，把 DeepSeekMoE 的配方变成了如今更接近行业默认设置的样子：用 sigmoid 门控取代 softmax；用一种免辅助损失的负载均衡方法（给每个专家的路由打分加一个可学习的偏置项，根据最近的负载动态上调或下调，而不是另设一个和语言建模目标互相竞争的损失项）；再加上一个和主模型一起训练的多 token 预测（MTP）模块。

还有两个更晚近、范围更窄的想法，虽然不是 DeepSeekMoE 这条主线的直系祖先，但对后面的内容同样重要：**Hash Layers，Roller 等人，2021 年** 证明了完全可以用一个固定的、确定性的哈希函数来路由 token，而不需要任何可学习的门控——用路由质量换取绝对均衡的负载和零路由网络开销。而 **Mixture-of-Depths，2024 年** 让 transformer 可以根据一个逐 token 学到的决策，选择让 token 穿过或绕过整个计算模块，是"让部分 token 跳过真正计算"这个想法最直接的概念祖先，此外还有另一项 2024 年提出的、用于加速专家并行通信的技术 ScMoE。

## 三、门控函数：从 softmax 到 sigmoid 及其变体

路由器最核心的工作，是把一个 token 的表示转换成对每个专家的打分，而用哪种函数来打分，已经悄悄变成了整个架构里影响最大的设计选择之一。早期的 MoE 工作——Shazeer、GShard、Switch Transformer、Mixtral——用的都是跨所有候选专家的 softmax，这会迫使每个专家在一个共享的概率预算里互相竞争：抬高一个专家的分数，必然会压低另一个的。DeepSeek-V3 换成了 sigmoid，给每个专家独立打分，不再互相竞争——这个改动在专家数量涨到成百上千、而不是维持在个位数的时候，影响会大得多。

sigmoid 这个选择基本上延续了下来：[GLM-5.2](https://www.zhipuai.cn/zh/research/161) 原封不动地保留了它，[DeepSeek-V4](https://arxiv.org/pdf/2606.19348) 也保持了同样的"独立打分"思路，只是把具体的函数换成了 Sqrt(Softplus)——论文里说这相较 V3 的 sigmoid 只是一个细微调整，不是性质上的改变。[MAI-Thinking-1](https://microsoft.ai/pdf/mai-thinking-1.pdf) 是唯一一个明显的反例，回退到了普通的 softmax，而且是在 token 原始的、未压缩的表示上计算的，尽管真正的专家计算发生在压缩后的潜空间里（后面会细讲）——值得注意的是，MAI-Thinking-1 也是这里唯一一个没有让每一层都用 MoE 的架构，而是把稠密前馈块和稀疏 MoE 块交替排布，这就带来一种可能性：门控函数的选择，或许在某种程度上和 MoE 在整个网络里被应用得有多密集是耦合在一起的，而不是一个放之四海而皆准的改进。[Kimi K3](https://www.kimi.com/blog/kimi-k3) 的门控函数，在其发布材料里没有披露。

## 四、负载均衡：从辅助损失到偏置项，再到分位数

路由器的训练目标本身并不会自然而然地阻止它把所有 token 都送去同样那几个专家、然后把其余专家全部闲置——这种失效模式通常被称为路由坍缩（routing collapse）——而如何在不伤害模型质量的前提下防止它，是 MoE 设计里过去两年被重新发明次数最多的部分。

GShard 最早的方案很直接：在训练目标上加一个辅助损失，惩罚不均衡的专家使用，后来又被 Switch Transformer 和 ST-MoE 的 router z-loss 进一步改进以稳定 logits。DeepSeek-V3 完全换了一种机制：给每个专家的路由打分加一个可学习的偏置项，专家用得不够时调高、用得太多时调低，这样均衡负载的同时不需要和语言建模损失去竞争梯度信号——所以叫"免辅助损失"。DeepSeek-V4 保留了这个偏置机制，但在此之上又叠加了一个小的序列级均衡损失，专门用来抓住单条长序列内部的不均衡——这种不均衡只有在按整个训练批次平均之后才会看起来正常，考虑到 V4 面向百万 token 上下文的目标，这一点尤其相关。

MAI-Thinking-1 走了另一个方向，用的是相当传统的 GShard 式辅助损失，而不是偏置项——但论文报告说，一旦这个损失在整个训练批次范围内做全局聚合，而不是按 micro-batch 分别计算，它的表现就和基于偏置项的方法差不多，这说明聚合策略本身可能比具体用哪种均衡公式更重要。Kimi K3 是离 DeepSeek-V3 默认设置最远的一个：它完全不用偏置项，而是用了 Moonshot 称之为 Quantile Balancing 的方法，直接从路由器自身打分分布的分位数推导出专家分配——同时消除了启发式的偏置更新规则和一个需要单独调节的均衡超参数。在 Kimi K3 896 选 16 这种稀疏度下，微小的路由偏差比在 DeepSeek-V3 256 选 8 下有更大的累积空间，所以一家前沿实验室想少一个容易调错的超参数，是很合理的诉求。同一个夏天，四种不同的均衡机制同时在发布，而公开材料里没有一份能干净、受控地证明其中哪一种全面胜过其他几种——很大程度上是因为每一种都是在一个特定的架构里调出来的（不同的粒度、不同的门控函数、不同程度的 dropless），单凭公开报告很难从中提炼出一个真正对等的比较。

## 五、粒度与共享专家问题

DeepSeekMoE 那套"细粒度专家 + 共享专家"的组合，已经不再是一个被激烈讨论的设计选择了——它是这篇文章里几乎所有内容共同的出发点。背后的逻辑很直接：大量的小专家，比少量的大专家能给路由器在给一个 token 分配容量时多得多的组合灵活性；而把一个共享专家隔离出来、让它无条件运行，意味着通用的、大家都需要的计算，不必在几十个本身也在努力做专精的路由专家里被反复重新学习。

Kimi K3 把"细粒度"这半边的想法推得比这篇文章里任何其他模型都更远——896 个专家，选 top-16——它的发布材料确认架构里存在共享专家组件（在它自己的架构图里可以看到），但没有公布具体数量。MAI-Thinking-1 是一个很有意思的反例：团队测试过共享专家之后发现，它只在每一层都用 MoE 的架构里才有帮助。由于 MAI-Thinking-1 是把稠密层和稀疏层交替排布，而不是这样做，共享专家在他们的消融实验里没有带来可测量的收益，最终发布的模型也没有用它——这提醒我们"标准配方"只是一个默认选项，而不是一条定律，而且这条默认设置本身是和"每一层都用 MoE"绑定在一起的。[LongCat-2.0](https://longcat.chat/blog/longcat-2.0/) 则完全走了另一条结构路线，把路由专家划分成三个按任务划分的组——Agent、Reasoning、Interaction——按任务分别门控，而不是把整个专家池当成一组无差别的、可以互换的通才专家。

## 六、把计算本身压缩掉：LatentMoE

到目前为止提到的想法，都没有动专家计算本身，只是在它周围调整路由或均衡方式。LatentMoE 则直接改变了计算本身——它由 NVIDIA 在 2026 年年初率先提出，已经用在自家 Nemotron-3 Super 和 Ultra 模型上：在专家计算和 all-to-all 通信发生之前，先把每个 token 投影进一个更小的潜空间维度，之后再展开回来。由于单个专家的计算量和网络流量都按同样的压缩倍数缩小，省下来的预算可以在总成本不变的前提下，重新投入到更多的专家数量和更大的 top-K 上。

MAI-Thinking-1 直接采用了这个思路，从更早的 192 个专家 / top-4 配置成长到了 512 个专家、top-8——正是在 LatentMoE 让这个更大的配置在通信上变得负担得起之后才实现的——并且完全采用 dropless 路由（完全没有 token 丢弃容量上限这回事），论文说这个选择是因为一旦引入硬性容量上限，他们在均衡消融实验里就会看到与容量相关的不一致结果。Kimi K3 的"Stable LatentMoE"看起来是同一个潜空间压缩家族、被用在了更极致的粒度上——896 个专家、top-16——不过 Moonshot 自己的材料没有像 MAI-Thinking-1 的报告那样，公布具体的压缩倍数或专家扩展比例。

## 七、让计算本身变得自适应

以上所有想法依然假定一个固定的 top-K：不管是一个标点符号还是一个真正困难的推理步骤，每个 token 激活的专家数量都是一样的。LongCat-2.0 直接打破了这个假设。它的专家池里包含了美团称之为 Zero-Computation Expert 的东西——这些"专家"完全不做任何计算，只是原样把输入透传出去——路由器会对每个 token 决定要用多少个真正的专家、多少个零计算专家。一个简单的 token 可能几乎全部被路由到零计算专家，几乎不花什么算力；一个困难的 token 才会调用更多真正的专家算力。这是一种按 token 自适应的算力预算，而不是固定的 top-K，从概念上比 DeepSeekMoE 这条谱系更接近 Mixture-of-Depths 在 2024 年提出的"让 token 完全跳过计算模块"的想法——同一个想法，被用在了专家维度上，而不是深度维度上。

这也是唯一一个还留有真实、未解答的开放问题的部分：美团还没有公布这套设计具体的负载均衡机制，而零计算专家这个机制本身就带来一个明显的问题——在训练时，是什么阻止了路由器为了在简单 token 上把 loss 降到最低，而把所有东西都路由到零计算专家。目前公开的材料还没有回答这个问题。

## 八、系统层面：路由到底要花多少通信成本

每一个被路由的 token，都得实实在在地通过互联，"跑"到持有它所选专家的那块加速卡上——这个成本只会随着专家数量涨到成百上千而变得更紧迫（这篇[分布式训练文章](#/blog?id=distributed-training-for-dl)更详细地讲过这种 all-to-all 通信模式）。上面提到的好几个选择，都是从不同角度对同一个成本问题给出的回应。DeepSeek-V4 里 Hash 路由的复活——确定性、无需学习的门控，专门用在前三层 MoE 上，取代了原本的稠密 FFN 块——用路由质量换取了这几层里绝对均衡的负载和零路由网络通信开销。LatentMoE 压缩后的潜空间，直接缩小了 all-to-all 交换中需要搬运的数据量，这正是让 MAI-Thinking-1 和 Kimi K3 一开始就能负担得起更大专家池的那个杠杆。而 DeepSeek-V4 另一项系统层面的贡献——以 MegaMoE 之名开源的流水线式专家并行内核——把专家并行 MoE 的 dispatch、计算、combine 三个阶段按专家"波次"重叠执行，而不是顺序执行，相比一个可比的重叠方案 Comet 报告了明显的加速。LongCat 的 ScMoE 骨干同样是一个系统驱动的选择，一种专门用来提升专家并行吞吐量的 shortcut-connected 设计。这些都不像 sigmoid 门控或细粒度专家那样是质量层面的提升——它们是让这些质量提升真正能负担得起运行成本的工程手段。

另外，DeepSeek-V4 还引入了 Anticipatory Routing，这是一项训练稳定性技术，而非通信技术：当模型检测到 loss 尖峰（论文明确把这归因于 MoE 层内部的异常激活值）时，会开始用路由器自身权重稍微滞后的一份拷贝来计算路由决策，而不是用当前权重，把路由器的更新速度和模型其余部分的更新速度解耦，直到不稳定状态过去。

## 十、哪些已经收敛，哪些还在争论

细粒度路由专家加一个独立共享专家，在 DeepSeekMoE 发表之后大约两年内，就不再是一个被激烈讨论的选择了——它是这整篇文章的共同出发点。Sigmoid 系门控同样已经在几乎所有地方取代了 softmax，MAI-Thinking-1 回退到 softmax 是唯一的例外——而且这恰好也是唯一一个没有让每一层都用 MoE 的架构。

负载均衡仍然明显每隔几个月就被重新发明一次：一个可学习的按专家偏置项、同样的偏置项再加一个序列级损失、一个完全按全局批次聚合的传统辅助损失，以及完全不需要单独超参数的分位数推导分配——这四种不同的答案在同一个夏天同时发布，而公开记录里没有任何受控比较能说明哪一种更胜一筹。

目前最重要的一个分歧点，在于两种不同的、拉大总参数量和激活参数量之间差距的方式。LatentMoE 保留了静态的 top-K，但通过把计算搬进压缩后的潜空间，降低了更大的 K 和更大的 N 的实际成本——专家更多、激活的也更多，但每一个都更便宜。LongCat-2.0 的零计算专家则保留了专家池名义上的大小不变，但让真正被激活的"实体"专家数量变成了一个动态的、取决于 token 的量。这两条路并不互斥，如果未来某份报告把两者结合起来——用潜空间压缩让更大的专家池变得负担得起，再用按 token 自适应的激活方式避免把这份预算平均花在不需要它的 token 上——也不会让人意外。就像任何针对一个快速演进的架构所做的快照一样，这里读起来像是已经收敛的部分，六个月后也可能会变得不一样。
