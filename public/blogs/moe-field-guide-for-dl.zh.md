今年夏天几周之内，五份技术报告接连发布：Moonshot AI 的 [Kimi K3](https://www.kimi.com/blog/kimi-k3)、智谱的 [GLM-5.2](https://www.zhipuai.cn/zh/research/161)、微软 AI 的 [MAI-Thinking-1](https://microsoft.ai/pdf/mai-thinking-1.pdf)、DeepSeek-AI 的 [DeepSeek-V4](https://arxiv.org/pdf/2606.19348)，以及美团的 [LongCat-2.0](https://longcat.chat/blog/longcat-2.0/)。这五个都是 Mixture-of-Experts（MoE）语言模型，但在路由、负载均衡、专家粒度这些问题上，五家给出的答案各不相同——而这些选择都不是凭空出现的。这篇文章讲的是一层现代 MoE 到底是怎么运作的、这五份报告里的每一个设计选择究竟能追溯到哪里，以及这个领域里哪些东西已经收敛成共识、哪些还在被激烈争论。

先说一下信息来源的可信度问题：下面 Kimi K3 和 DeepSeek-V4 的数字直接来自它们各自的发布材料；MAI-Thinking-1 的数字来自对微软技术报告的完整阅读。GLM-5.2 自己的发布页面在 MoE 架构具体数字上着墨很少（重点放在 coding 基准测试和基础设施上），所以这里给出的专家数量和 top-K 数字是从第三方技术报道中整理出来的，而非智谱自己的文字——请把它们当作来源可靠但属于二手信息。LongCat-2.0 的博客页面在这次研究中没能直接抓取到渲染后的架构细节，所以它的数字同样来自第三方报道。凡是二手来源而非模型自身报告直接确认的数字，都会在文中标注清楚。

## 一、一层现代 MoE 长什么样

把所有厂商特有的细节都去掉，2026 年这一代的 MoE 层大致是这样运作的：一个 token 进来，一个小型路由网络给每个专家打一个"匹配度"分数，得分最高的 top-K 个专家真正跑一遍前馈计算，而且——在现在几乎所有的设计里——还有一个额外的共享专家，不管路由器怎么判断，它都会对每个 token 无条件运行。所有输出按路由器给出的权重加权合并，就是这一层的输出。

![一层现代 MoE 的解剖图](blogs/images/moe-layer-anatomy.svg?v=1)
^一个 token，一层：每个路由专家都是候选项，真正跑起来的只有寥寥几个——共享专家是唯一的例外，它永远会被激活。

这个设计的全部魅力都藏在两个数字之间的落差里。**总参数量**取决于专家池里有多少专家——这就是厂商放进头条的数字（"2.8 万亿参数"）。**激活参数量**取决于每个 token 真正跑了多少专家——top-K 加上共享专家——这才是真正决定推理成本的数字。只要 K 相对于整个专家池保持得足够小，一个总参数量巨大的模型，单个 token 的推理成本依然可以很低。总量和激活量之间的这道落差，就是 MoE 存在的全部理由，而这篇文章里提到的每一个设计选择，说到底都是在尝试把这道落差拉得更大，同时又不伤害模型质量。

有两个机制上的细节值得在这里单独说一下，因为下面每一份报告都对它们做出了明确的选择。

**门控函数。** 路由器需要把一个 token 的表示转换成对每个专家的打分。早期的 MoE 工作用的是跨所有专家的 softmax（迫使专家们在一个固定的概率预算里互相竞争）；更新的设计则改用 sigmoid，给每个专家独立打分——当你要从成百上千个专家里选出几十个而不是从八个里选出两三个时，这个区别就变得重要起来。

**负载均衡。** 路由器的训练目标本身并不会自然而然地阻止它把所有 token 都送去同样那三个专家、然后把其余专家全部闲置——这种失效模式通常被称为路由坍缩（routing collapse）。这里提到的每一个设计都有某种机制来防止它，而具体用哪种机制，在过去两年里的变化比这个领域里几乎任何其他东西都要大。

## 二、这些设计到底从哪来：九年的谱系

2026 年这五份报告没有一份是 Mixture-of-Experts 的发明者，其中大部分看起来"新"的东西，要么是某个更老想法的变体，要么是对某一篇 2024 年论文的直接延续。

![MoE 设计的九年演变](blogs/images/moe-lineage-timeline.svg?v=1)
^2026 年大多数前沿 MoE 报告都能追溯到 2024 年的同一个分叉点：DeepSeekMoE 的"细粒度专家 + 共享专家"配方。

**Shazeer 等人，2017 年**为深度学习引入了现代意义上的稀疏门控 MoE 层：一个带噪声的 top-k softmax 路由器，配合一个辅助损失来维持专家使用的均衡。这基本上是后面所有工作的共同祖先。

**GShard（2020 年）**把这个想法扩展到了真正跨硬件做专家并行训练的 transformer 上——把专家切分到不同设备，用 all-to-all 通信把 token 送到持有其对应专家的设备。标准的负载均衡辅助损失公式也是从这里来的，容量因子（每个专家能接收的 token 数硬上限，超出部分会被丢弃）也是从这里进入标准工具箱的。

**Switch Transformer 和 GLaM（2021 年）**从同一个基础出发走向了不同方向：Switch Transformer 把路由简化成 top-1（每个 token 只用一个专家），证明了即便在这种极端设置下 MoE 依然能稳定训练；GLaM 则用 top-2 路由把参数规模做到 1.2 万亿，每个 token 只激活其中很小一部分——这是"把总参数量和计算成本解耦"这件事一次早期而干净的示范。

**ST-MoE 和 expert-choice 路由（2022 年）**解决了一个训练稳定性问题（路由器的 logits 可能无限增长导致训练不稳定，用一个 router z-loss 来解决），并探索了一套完全不同的范式——不再是每个 token 挑选它的 top-K 专家，而是让每个专家挑选属于它的 top-tokens，这样从结构上就保证了完美的负载均衡，但也让因果/自回归生成变得更复杂。

**Mixtral（2024 年）**与其说带来了新的 MoE 机制，不如说证明了整套方法可以被训练出来、并作为一个强力的开源权重模型发布——采用相当标准的 softmax 路由器，做 8 选 2 的路由——从此让 MoE 在任何想做开源模型的团队眼里，变成了一个主流、不再显得另类的架构选择。

**DeepSeekMoE（2024 年）**才是后面几乎所有内容真正的分叉点。它的两个贡献是：第一，细粒度专家切分——不用少量的大专家，而是用大量的小专家，这给了路由器在给一个 token 分配容量时多得多的组合灵活性；第二，把一个或多个共享专家隔离出来，让它们无条件处理每一个 token，这样通用的、大家都需要的计算就不必在几十个路由专家里被反复重复学习。这篇文章覆盖的五份报告里，有四份要么直接引用了这篇论文，要么建立在沿用它的谱系之上。

**DeepSeek-V3（2024 年末）**接着做出了三个改动，把 DeepSeekMoE 的配方变成了如今更接近行业默认设置的样子：用 sigmoid 门控取代 softmax；用一种免辅助损失的负载均衡方法（给每个专家的路由打分加一个可学习的偏置项，根据最近的负载动态上调或下调，而不是另设一个和语言建模目标互相竞争的损失项）；再加上一个和主模型一起训练的多 token 预测（MTP）模块。这一整套组合——细粒度路由专家、一个共享专家、sigmoid 门控、基于偏置项的均衡——正是 GLM-5.2 和 DeepSeek-V4 在 2026 年依然在用、几乎原封不动的配方。

还有两个更晚近、范围更窄的想法，虽然不是 DeepSeekMoE 这条主线的直系祖先，但对后面的内容同样重要：**Hash Layers（Roller 等人，2021 年）**证明了完全可以用一个固定的、确定性的哈希函数来路由 token，而不需要任何可学习的门控——用路由质量换取绝对均衡的负载和零路由网络开销——这个想法在 DeepSeek-V4 里被用于一个很窄的具体场景，重新复活了。而**Mixture-of-Depths（2024 年）**让 transformer 可以根据一个逐 token 学到的决策，选择让 token 穿过或绕过整个计算模块，是 LongCat-2.0 里"让部分 token 跳过真正计算"这个想法最直接的概念祖先，此外 LongCat 的骨干网络还直接建立在另一项 2024 年提出的、用于加速专家并行通信的技术 ScMoE 之上。

## 三、DeepSeek-V4：直系继承者，打磨而非重造

[DeepSeek-V4](https://arxiv.org/pdf/2606.19348) 发布了两个模型——DeepSeek-V4-Flash（总参数 2,840 亿，激活 130 亿）和 DeepSeek-V4-Pro（总参数 1.6 万亿，激活 490 亿）——两者都基本完整保留了 DeepSeekMoE 那套细粒度加共享专家的框架：Flash 是 256 个路由专家 + 1 个共享专家，Pro 是 384 个路由专家 + 1 个共享专家，两者都是 top-6。论文里明确说这些相较 V3 只是"细微调整"，而这些调整确实是渐进式的，而非结构性的：打分函数从 V3 的 sigmoid 换成了 Sqrt(Softplus)，免辅助损失的偏置均衡机制之上又叠加了一个小的序列级均衡损失，专门用来防止单条长序列内部的不均衡（考虑到 V4 面向百万 token 上下文的目标，这一点尤其相关）——哪怕整个训练批次平均下来看均衡情况是正常的。

更有意思的是两个范围更窄的改动。第一，前三层 MoE 从可学习路由器换成了 Hash 路由——确定性、无需梯度、也不可能坍缩——是对 Roller 等人 2021 年那个想法的直接复活，用在了 DeepSeek-V3 曾经放稠密（非 MoE）层的位置上。第二，一项名为 Anticipatory Routing 的训练稳定性技术：当模型检测到 loss 尖峰（论文明确把这归因于 MoE 层内部的异常激活值）时，会开始用路由器自身权重稍微滞后的一份拷贝来计算路由决策，而不是用当前权重，把路由器的更新速度和模型其余部分的更新速度解耦，直到不稳定状态过去。工程层面，DeepSeek-V4 还引入了一种新的流水线式专家并行内核（以 MegaMoE 之名开源，是 DeepGEMM 的一部分），把专家并行 MoE 的 dispatch、计算、combine 三个阶段按专家"波次"重叠执行，相比一个可比的重叠方案 Comet 报告了明显的加速。

## 四、GLM-5.2：把这套配方用成了行业默认设置

根据第三方技术报道，[GLM-5.2](https://www.zhipuai.cn/zh/research/161) 是一个总参数 7,530 亿、激活 400 亿的模型，用 256 个路由专家、top-8，加 1 个共享专家，sigmoid 门控，DeepSeekMoE 式的免辅助损失均衡——也就是说，基本上就是 DeepSeek-V3 那套 MoE 配方，几乎没有变化。这不是在批评它，反而是更值得关注的一点。智谱自己的发布材料几乎不怎么谈 MoE 架构本身，而是把重点放在了 IndexShare（一种注意力侧的优化，每四层共享同一个稀疏注意力索引器，在长上下文场景下把单位 token 的 FLOPs 降低到约 2.9 倍）、一个为投机解码改进过的多 token 预测层（把接受长度最多提升 20%），以及 Slime——一套用于大规模 agentic 强化学习的自研框架。到 2026 年，"细粒度专家 + 共享专家 + sigmoid + 免辅助损失均衡"这套组合已经足够稳定、足够被充分理解，以至于一家前沿实验室可以把它当成一层"已解决"的问题，把差异化的预算完全花在别的地方。

## 五、MAI-Thinking-1：拥抱 LatentMoE，砍掉共享专家

微软 AI 的 [MAI-Thinking-1](https://microsoft.ai/pdf/mai-thinking-1.pdf)（激活 347 亿 / 总参数 9,620 亿，78 层）是这五份报告里第一个真正重构了 MoE 层本身、而不只是调整均衡机制的模型。它采用了 LatentMoE——这是 NVIDIA 在 2026 年年初率先提出、并已经用在自家 Nemotron-3 Super 和 Ultra 模型上的一种架构——在专家计算和 all-to-all 通信发生之前，先把每个 token 压缩进一个更小的潜空间维度，之后再展开回来。由于单个专家的计算量和网络流量都按同样的压缩倍数缩小，省下来的预算被重新投入到更多的专家数量和更大的 top-K 上：MAI-Thinking-1 跑的是 512 个专家、top-8，这是从更早的 192 个专家 / top-4 配置成长而来的——正是在 LatentMoE 让这个更大的配置在通信上变得负担得起之后才实现的。

MAI-Thinking-1 的另外三个选择值得单独一提，因为它们和 DeepSeek-V3 的默认设置正好相反。它用的是普通的 softmax 门控，而且是在 token 原始的、未压缩的表示上计算的，尽管真正的专家计算发生在压缩后的潜空间里。它的负载均衡损失是一种相当传统的 GShard 式辅助损失，而不是基于偏置项的免均衡项——但论文报告说，一旦这个损失在整个训练批次范围内做全局聚合，而不是按 micro-batch 分别计算，它的表现就和基于偏置项的方法差不多——这说明聚合策略本身可能比具体用哪种均衡公式更重要。而且，团队测试过共享专家之后发现，它只在每一层都用 MoE 的架构里才有帮助；MAI-Thinking-1 是把稠密前馈层和稀疏 MoE 层交替排布，而不是让每一层都稀疏，在这种交替结构下，共享专家没有带来可测量的收益——所以最终发布的模型没有用它。这个模型还完全采用 dropless 路由（完全没有 token 丢弃容量上限这回事），论文说这个选择是因为一旦引入硬性容量上限，他们在均衡消融实验里就会看到与容量相关的不一致结果。

## 六、Kimi K3：把粒度推向极致，换掉均衡项

[Kimi K3](https://www.kimi.com/blog/kimi-k3) 是这次对比里规模最大的模型，总参数 2.8 万亿（Moonshot 没有公布激活参数的具体数字），而且它把专家粒度推得比 2026 年任何其他报告都更细：896 个专家，每个 token 选 top-16，放在 Moonshot 称之为 Stable LatentMoE 的框架里——看起来和 MAI-Thinking-1 的 LatentMoE 属于同一个潜空间压缩家族，只是被用在了更极致的粒度上。发布博客确认架构里存在共享专家组件（在它自己的架构图里可以看到），但没有公布具体数量。

更新颖的地方在于它处理负载均衡的方式。它没有用 DeepSeek-V3 那种可学习的偏置项，而是用了 Moonshot 称之为 Quantile Balancing 的方法：专家分配直接从路由器自身打分分布的分位数推导出来，发布材料里说这样做消除了启发式的偏置更新规则，也不再需要一个单独调节的负载均衡超参数。在 16 选 896 这种稀疏度下，微小的路由偏差比在 DeepSeek-V3 的 8 选 256 下有更大的累积空间，所以一家前沿实验室想少一个容易调错的超参数，是很合理的诉求。Kimi K3 还引入了 Per-Head Muon（把 Muon 优化器扩展成对注意力头分别独立优化），并且报告说从 SFT 阶段开始就采用了量化感知训练，用 MXFP4 权重配合 MXFP8 激活值。

## 七、LongCat-2.0：把专家池本身的形状都改了

[LongCat-2.0](https://longcat.chat/blog/longcat-2.0/)（总参数 1.6 万亿，平均激活约 480 亿——据报道会在 330 亿到 560 亿之间浮动）在结构上是这里面最不一样的一个。这篇文章里的其他每个模型都保留了某个版本的"N 个路由专家，选 top-K"这种固定形状；LongCat 则让这个形状按 token 变化。它的专家池里包含了美团称之为 Zero-Computation Expert 的东西：这些"专家"完全不做任何计算，只是原样把输入透传出去。路由器会对每个 token 决定要用多少个真正的专家、多少个零计算专家——一个简单的 token（比如标点符号）可能几乎全部被路由到零计算专家，几乎不花什么算力；而一个困难的 token 才会调用更多真正的专家算力。这是一种按 token 自适应的算力预算，而不是固定的 top-K，从概念上更接近 Mixture-of-Depths 里"让部分 token 跳过计算"的想法，只是把它用在了专家维度上，而不是深度维度上。

除此之外，LongCat 的 MoE 骨干采用了一种 shortcut-connected 设计（ScMoE），专门用来提升专家并行的吞吐量；路由专家本身还被划分成三个按任务划分的组——Agent、Reasoning、Interaction——路由是按任务分别门控的，而不是把整个专家池当成一组无差别的、可以互换的通才专家。另外还有一个独立的、1,350 亿参数的 n-gram 嵌入模块，完全在 MoE 堆栈之外单独存在。美团还没有公布这套设计具体的负载均衡机制，这在一份原本相当详尽的发布材料里算是一个真正的空白——尤其是零计算专家这个机制本身就带来一个明显的均衡问题（在训练时，是什么阻止了路由器为了在简单 token 上把 loss 降到最低，而把所有东西都路由到零计算专家），目前公开的材料还没有回答这个问题。

## 九、哪些已经收敛，哪些还在争论

细粒度路由专家加一个独立共享专家，已经不再是一个被激烈讨论的设计选择了——它是这篇文章里每一个模型共同的出发点，DeepSeekMoE 在 2024 年的贡献，在大约两年内就变成了标配。Sigmoid 系门控（或者像 Sqrt(Softplus) 这样的近亲变体）同样已经在这里除 MAI-Thinking-1 之外的每个模型里取代了 softmax——而 MAI-Thinking-1 恰好也是唯一一个没有让每一层都用 MoE 的架构，这说明门控函数的选择，或许和 MoE 在整个网络里被应用得有多密集存在某种耦合关系，而不是一个放之四海而皆准的改进。

负载均衡是仍然明显每隔几个月就被重新发明一次的部分：一个可学习的按专家偏置项（DeepSeek-V3、GLM-5.2）、同样的偏置项再加一个序列级损失（DeepSeek-V4）、一个完全按全局批次聚合的传统辅助损失（MAI-Thinking-1）、以及完全不需要单独超参数的分位数推导分配（Kimi K3）——同一个夏天，四种不同的答案同时在发布。公开材料里没有一份能干净、受控地证明其中哪一种全面胜过其他几种——很大程度上是因为每一种都是在一个特定的架构里调出来的（不同的粒度、不同的门控函数、不同程度的 dropless），单凭公开报告很难从中提炼出一个真正对等的比较。

目前最重要的一个分歧点，在于两种不同的、拉大总参数量和激活参数量之间差距的方式。LatentMoE（MAI-Thinking-1、Kimi K3）保留了静态的 top-K，但通过把计算搬进压缩后的潜空间，降低了更大的 K 和更大的 N 的实际成本——专家更多、激活的也更多，但每一个都更便宜。LongCat-2.0 的零计算专家则保留了专家池名义上的大小，但让真正被激活的"实体"专家数量变成了一个动态的、取决于 token 的量——精神上和 Mixture-of-Depths 是同一个想法，只是被用在了专家维度上，而不是深度维度上。这两条路并不互斥，如果 2027 年的某份报告把两者结合起来——用潜空间压缩让更大的专家池变得负担得起，再用按 token 自适应的激活方式避免把这份预算平均花在不需要它的 token 上——也不会让人意外。

## 十、如何读懂一张 MoE 规格表

**预测单个 token 推理成本的是激活参数量，不是总参数量**——一个总参数 2.8 万亿、但 top-K 很小的模型，跑起来完全可能比一个 3,000 亿的稠密模型更便宜，总参数量之所以常常被放进标题，恰恰是因为它是更唬人的数字，而不是更能反映实际运行成本的数字。

**共享专家不是"路由器每次都投票给它的那个路由专家"**——它在架构上是完全独立的一份权重，从来不参与 top-K 的竞争。它存在与否本身就是一个设计选择，而不是一个无关紧要的细节：MAI-Thinking-1 最终没有用共享专家，恰恰是因为它交替排布稠密和稀疏层的结构让共享专家变得多余——这提醒我们"标准配方"只是一个默认选项，而不是一条定律。

**专家池变大，增加的不只是权重存储，还有通信成本**——每一个被路由的 token 都得实实在在地通过互联，"跑"到持有它所选专家的那块加速卡上（这篇[分布式训练文章](#/blog?id=distributed-training-for-dl)更详细地讲过这种 all-to-all 通信模式）。DeepSeek-V4 里 Hash 路由的复活，以及 LatentMoE 被压缩过的通信负载，都是从不同角度对同一个成本问题给出的回应——而这个成本只会随着专家数量涨到成百上千而变得更紧迫。

**没有公开的数字就当作没有公开，不要当成零，也不必觉得这是什么难堪的空白**——Kimi K3 的博客对激活参数量和共享专家的具体数量确实只字未提，LongCat-2.0 零计算专家的均衡机制目前也还没有公开。前沿实验室会披露那些对自己定位有利的信息，仅此而已；一个缺失的数字，反映的是披露策略本身的事实，未必是架构本身的事实。

*本文中的技术论述，在完整抓取到原始材料的地方（Kimi K3、MAI-Thinking-1、DeepSeek-V4）直接来自各模型自己的发布材料；在原始来源没有提供架构层面 MoE 细节的地方（GLM-5.2、LongCat-2.0），则来自第三方技术报道，均已在正文中标注。和任何针对同时期发布的报告所做的横向比较一样，这里读起来像是已经收敛的部分，六个月后也可能会变得不一样。*
